## 1. What MeMo Does (Executive Summary)

MeMo is a **modular knowledge integration framework** that separates **memory** from **reasoning**:

| Component | Role | Size | Trainable? |
|-----------|------|------|------------|
| **GENERATOR** | Synthesizes a "reflection QA dataset" from raw corpus | Same or smaller than EXECUTIVE | Frozen |
| **MEMORY model** | Internalizes corpus knowledge parametrically | 1.5B–14B parameters | **Trained via SFT** |
| **EXECUTIVE model** | Reasons over MEMORY responses, answers user queries | 32B+ parameters (any LLM) | **Frozen** |

**Key insight:** Instead of retrieving document chunks at inference (RAG) or fine-tuning the LLM itself (parametric), MeMo trains a **dedicated small model** to act as a memory oracle. The EXECUTIVE queries it via a structured multi-turn protocol.

### The Five-Step Data Synthesis Pipeline

```
Raw Corpus D
    ↓
[1] Fact Extraction → Qdir (explicit facts) + Qindir (inferred facts)
    ↓
[2] Consolidation → Qmrg (merged multi-fact pairs)
    ↓
[3] Verification & Rewriting → Qver (self-contained QA pairs)
    ↓
[4] Entity Surfacing → Qent (entity-identification QA pairs)
    ↓
[5] Cross-Document Synthesis → Qcross (multi-document relationships)
    ↓
Qfinal = Qver ∪ Qent ∪ Qcross → trains MEMORY model
```

### The Three-Stage Inference Protocol

```
User Query q
    ↓
[Stage 1: Grounding] EXECUTIVE decomposes q → atomic sub-questions → MEMORY answers
    ↓
[Stage 2: Entity Identification] EXECUTIVE narrows candidates via follow-up queries to MEMORY
    ↓
[Stage 3: Answer Synthesis] EXECUTIVE queries MEMORY for supporting facts → synthesizes final answer
```

## 2. Why MeMo Matters for Medha

### 2.1 The Construction Document Problem MeMo Solves

Our BFS research (BFS-014 through BFS-018) identified that construction document coordination suffers from **cross-document relationship blindness**:

| Current RAG Limitation | MeMo Solution | Medha Application |
|------------------------|---------------|-------------------|
| Retrieves isolated chunks; misses relationships between spec §23 31 13 and Drawing A-101 | **Cross-document synthesis** (Step 5) creates QA pairs that require multi-document reasoning | MEMORY model internalizes "spec ↔ drawing ↔ RFI ↔ clash report" relationships |
| Noisy retrieval returns irrelevant chunks; 60% of affected documents missed during change orders [BFS-018] | **Robust to retrieval noise** (Table 3) — MEMORY answers from parametric knowledge, not raw chunks | Contradiction detection avoids false positives from irrelevant retrieved context |
| Context window limits prevent feeding entire spec + all drawings into prompt | **Retrieval cost independent of corpus size** — MEMORY model is fixed-size | Medha scales from one project to portfolio without inference slowdown |
| Fine-tuning Grok/Claude/GPT-4 is impossible (proprietary) or expensive | **Plug-and-play with black-box LLMs** — only MEMORY is trained; EXECUTIVE is any API | Medha can use best-in-class reasoning models without vendor lock-in |
| New drawings/specs arrive weekly; retraining everything is expensive | **Continual integration via model merging** (Sec 4.3, Table 6) — merge new MEMORY models without full retraining | New project documents added incrementally; 33% compute savings at K=2 |

### 2.2 The "Reflection" Concept Applied to Construction

MeMo's core innovation is the **reflection** — a compositional representation that exposes corpus knowledge under diverse query variations.

**For construction documents, a reflection might look like:**

> **Q:** "What fire rating is required for the MEP riser shaft on Level 7, and which drawing detail governs it?"
> **A:** "The MEP riser shaft on Level 7 requires a 2-hour fire rating per NFPA 13 Section 8.15.5, as specified in Fire Protection Spec FPS-211313 §2.3.A. The governing drawing detail is A-701, which shows the shaft enclosure with 2-hour rated gypsum board. Note: Drawing A-701 (Rev C) conflicts with Spec §2.3.A because the detail shows 1-hour rating — this discrepancy was flagged in Clash Report CL-042 and RFI-284 remains unresolved."

This single reflection captures:
- Cross-document relationship (spec ↔ drawing)
- Code reference (NFPA 13)
- Version conflict (Rev C shows wrong rating)
- Active issue linkage (CL-042 → RFI-284)

**Current Medha RAG cannot generate this** because it retrieves chunks independently and has no mechanism to synthesize cross-document contradictions.

### 2.3 MeMo vs. Medha's Current Architecture

```
Current Medha (RAG-based):
  User Query → Embed → Retrieve chunks from Chroma/pgvector →
  Stuff into LLM prompt → Generate answer

  Problem: Chunks are isolated. Context window is limited.
  Retrieval noise degrades quality.

MeMo-enhanced Medha:
  User Query → EXECUTIVE (Grok/Claude/GPT-4) decomposes →
  Sub-queries to MEMORY model (trained on construction reflections) →
  MEMORY answers from parametric knowledge →
  EXECUTIVE synthesizes final answer

  Advantage: MEMORY internalizes cross-document relationships.
  No retrieval noise. Scales to large corpora. Works with any LLM.
```

## 3. Specific Medha Features MeMo Enables

### 3.1 P0-1: Contradiction Detection

**Current approach:** RAG retrieves spec chunks and drawing chunks separately; LLM compares them in-context.

**MeMo approach:** During training, Step 5 (Cross-Document Synthesis) explicitly generates QA pairs that require identifying contradictions across documents. The MEMORY model learns to flag mismatches parametrically. At inference, the EXECUTIVE asks: *"Does spec §23 31 13 agree with drawing A-101 on material specification?"* — MEMORY answers with the contradiction directly, no retrieval needed.

**Impact:** Eliminates false positives from retrieval noise. Captures contradictions that span >2 documents.

### 3.2 P0-2: Semantic Document Search

**Current approach:** Vector similarity search over embedded chunks.

**MeMo approach:** The MEMORY model itself becomes the search interface. Instead of "find chunks similar to query," the EXECUTIVE asks MEMORY targeted questions: *"Which spec sections govern MEP ductwork on Level 7?"* MEMORY answers from internalized knowledge, returning precise references rather than semantically similar but potentially irrelevant chunks.

**Impact:** Search precision improves; no dependency on embedding quality or chunk size.

### 3.3 P0-3: RFI Auto-Draft

**Current approach:** Detect contradiction → retrieve related docs → prompt LLM to draft RFI.

**MeMo approach:** MEMORY already knows the contradiction, the spec clause, the drawing reference, and the clash report. EXECUTIVE asks: *"Draft an RFI for the contradiction between spec §23 31 13 and drawing A-101, citing clash report CL-042."* MEMORY provides the structured facts; EXECUTIVE formats them into professional RFI language.

**Impact:** Faster, more consistent RFI drafting with complete citation chains.

### 3.4 P1-3: Impact Analysis

**Current approach:** Manual cross-referencing to find affected documents during change orders.

**MeMo approach:** Cross-document synthesis during training creates explicit "affected by" relationships in MEMORY's parametric knowledge. EXECUTIVE asks: *"If we change the fire rating in spec FPS-211313 §2.3.A, which drawings, RFIs, and submittals are affected?"* MEMORY returns the complete impact graph.

**Impact:** CMAA 2019 found manual cross-referencing misses 60% of affected documents [BFS-018]. MeMo internalizes these relationships.

### 3.5 P1-6: Cross-Platform Context Bridge

**Current approach:** RAG indexes documents from multiple sources (Navisworks exports, Procore RFIs, Aconex transmittals) but treats them as isolated chunks.

**MeMo approach:** The reflection dataset explicitly synthesizes cross-platform relationships: *"Clash CL-042 (from Navisworks) triggered RFI-284 (in Procore), which references spec §23 31 13 and drawing A-101."* MEMORY internalizes this as a single compositional fact.

**Impact:** Information that evaporates at platform boundaries (BFS-018 §4.1) is preserved in MEMORY's parametric knowledge.

## 4. Implementation Path for Medha

### 4.1 Phase 1: Proof of Concept (1–2 weeks)

**Goal:** Validate MeMo on a small construction document corpus.

1. **Corpus:** Select 5–10 construction documents (1 spec, 3–4 drawings, 2–3 RFIs, 1 clash report, 1 BEP) from our `sample_docs/` or `real_construction_docs/`
2. **GENERATOR:** Use GPT-4o or Claude 3.5 Sonnet to run the 5-step reflection synthesis pipeline
3. **MEMORY model:** Fine-tune Qwen2.5-1.5B-Instruct on the reflection QA dataset (small, fast)
4. **EXECUTIVE:** Use the same LLM (GPT-4o/Claude) in black-box mode
5. **Test queries:**
   - "What material is specified for HVAC ductwork on Level 7?"
   - "Does drawing A-101 agree with spec §23 31 13 on duct insulation?"
   - "Which RFIs are related to the MEP riser shaft?"
   - "What changed between drawing Rev B and Rev C?"

### 4.2 Phase 2: Scale to Dubai Corpus (4–6 weeks)

**Goal:** Train MEMORY model on a representative Dubai construction document set.

1. **Corpus:** Collect 50–100 documents (specs, drawings, RFIs, submittals, clash reports) from Dubai projects
2. **GENERATOR:** Qwen2.5-32B-Instruct (as in the paper) or GPT-4o
3. **MEMORY model:** Qwen2.5-14B-Instruct (as in the paper's best config)
4. **Training:** 3 epochs, DeepSpeed 2, ~24 GPU-hours on H100
5. **Evaluation:** Compare against current RAG (Chroma + embedding) on contradiction detection accuracy

### 4.3 Phase 3: Production Integration (8–12 weeks)

**Goal:** Replace or augment Medha's retrieval layer with MeMo MEMORY model.

1. **Architecture:**
   - Frontend → Backend API → EXECUTIVE (Grok/Claude/GPT-4)
   - EXECUTIVE queries MEMORY model via multi-turn protocol
   - MEMORY model served via vLLM or similar inference server
2. **Continual integration:** As new documents arrive per project, generate reflection QA, train new MEMORY model, merge with existing via TIES (ρ=0.3)
3. **Monitoring:** Track contradiction detection precision/recall, RFI drafting quality, user satisfaction

## 5. Key Results from the Paper (Relevant to Medha)

| Benchmark | What It Tests | MeMo (Qwen 32B Exec) | Best RAG Baseline | Gap |
|-----------|--------------|----------------------|-------------------|-----|
| **NarrativeQA** | Long-document reasoning | **26.85%** | HippoRAG2: ~15% | +11% |
| **NarrativeQA** | Long-document reasoning | **53.58%** (Gemini exec) | — | — |
| **MuSiQue** | Multi-hop (2–4 steps) | **48.30%** | HippoRAG2: ~42% | +6% |
| **BrowseComp-Plus** | Deep research, multi-doc | **54.22%** | HippoRAG2: 56.11% | -2% |
| **BrowseComp-Plus** | Deep research, multi-doc | **66.67%** (Gemini exec) | — | — |

**Noise robustness (Table 3):**
- HippoRAG2 drops **6.22%** on BrowseComp-Plus when noise increases 0× → 1×
- MeMo changes by only **+0.55%** (essentially flat)
- This is critical for Medha because construction document corpora are inherently noisy (outdated revisions, unrelated specs, distractor drawings)

**Model merging for continual integration (Table 6):**
- Full retrain: 72 GPU-hours cumulative
- Model merging (TIES, ρ=0.3): 48 GPU-hours — **33% savings**
- Merged model still outperforms all retrieval baselines

## 6. Risks and Limitations

| Risk | Paper's Admission | Medha Mitigation |
|------|------------------|------------------|
| Training cost | MEMORY model training is expensive (24–72 GPU-hours per corpus) | Start with 1.5B model; use model merging for incremental updates; train only once per project |
| MEMORY model capacity | May not scale to extremely large corpora (thousands of documents) | Partition by project; one MEMORY model per project; merge at portfolio level |
| Evaluation scope | Tested on QA benchmarks, not construction documents | Run Phase 1 PoC on real construction docs to validate |
| Dynamic corpora | Documents change frequently in construction | Model merging supports incremental updates; full retrain only on major milestones |

## 7. Recommendation

**MeMo is highly relevant to Medha's core technical challenge.** The paper directly addresses the cross-document relationship problem that our BFS research identified as the #1 limitation of current construction collaboration platforms.

**Specific recommendation:** Add MeMo as a **P1 architectural direction** in the PRD. Run a Phase 1 PoC (1–2 weeks, $50–100 in API costs) on our existing sample construction documents before committing to full implementation. If the PoC demonstrates improved contradiction detection and cross-document reasoning over our current RAG approach, prioritize MeMo for the retrieval substrate redesign.
